Why skills, really

Models are brilliant. They're missing your expertise.

Models are trained on the whole internet, so they're loaded with semantic knowledge, facts and data. What they don't have is your procedural knowledge, the workflows only you know. A Skill is how that gets loaded in.
Semantic knowledge

Facts & data

e.g. Kubernetes architecture. Models already have this, it's what they were trained on.

Procedural knowledge

Workflows

e.g. your team's 47-step release checklist. Models don't have this, it lives in your head. A Skill is how it gets loaded in.

LLMs excel at facts but lack specific workflows. That gap is exactly what a Skill fills.
How a Skill actually looks

One open standard, the same shape everywhere

Before comparing a Skill to anything else, here's what one actually is: a folder.
  • A SKILL.md file up top: name, description, instructions
  • Optional references/, scripts/ and assets/ alongside it, loaded only as needed
  • Same shape, runs unmodified in Claude Code, Antigravity, GitHub Copilot, Google ADK, even fully offline

Mechanics

Progressive disclosure prevents context bloat

Level 1: Metadata Always visible
name + description · ~100 tokens, loaded at startup for every skill
↓ task matches the description?
Level 2: Instructions Loaded once selected
full SKILL.md body · under 5,000 tokens recommended
↓ needs scripts, references or assets?
Level 3: Resources Fetched only if used
references/ · scripts/ · assets/ · loaded on demand, no token cost until then
Scales to 100s of capabilities while keeping the primary context window lean.
Live example: github.com/anthropics/skills, skill-creator, Anthropic's own skill for drafting, testing and validating skills against this exact spec.
Stat
~1,700 tokens
is roughly what 17 skills' worth of frontmatter costs, combined. An agent can be aware of dozens of skills for less context than loading a single one in full.

Where this is heading

Agents are starting to write their own skills

Google's ADK already has a “meta skill” pattern: ask the agent to build a skill for something, and it writes a new SKILL.md on the spot, following the same open agentskills.io spec, ready to reuse and commit to git like any other code.
Thousands of skills built since launch: foundational ones (document creation, scientific data analysis), partner integrations (Browserbase, Notion), and Fortune 100 companies standardising internal workflows, not just for developers, for their whole staff.
The vision: agents that capture what they learn as a reusable skill, and hand it to their future selves, or their teammates. Continuous learning, file by file.

A Skill vs a prompt library

A prompt library is text you copy and paste. A Skill is a folder the agent discovers and loads itself, only when it's relevant, no copy-paste required.
Quick compare, 2 of 5

A Skill vs MCP

MCP is the protocol: the tools the agent is allowed to call. A Skill is the procedure: how it uses them once it has them. Complementary, not alternatives, you'll often use both together.
Session 2 is the MCP session, today's accessibility demo already uses one under the hood. MCP was donated to the Agentic AI Foundation by Anthropic, same open-governance story as everything else today.
Quick compare, 3 of 5

A Skill vs RAG

RAG grounds the agent in fresh knowledge it retrieves. A Skill grounds it in a fixed procedure it already has. RAG answers “what do I know.” A Skill answers “what do I do.”

A Skill vs AGENTS.md

Think of AGENTS.md as a project-specific system prompt: always loaded, once, for the whole project. A Skill is loaded on demand, only for the task that needs it.

A Skill vs a Workflow

A workflow is a fixed sequence of steps, authored once, running the same way every time. A Skill isn't a pipeline: whether and how it's used depends on the agent reasoning about the task.
Not rivals: a workflow can call a Skill at any step, and an agent can use that same Skill inside its own loop. The difference is who decides how it's run, the workflow's author in advance, or the agent, in the moment.
Quick compare, bonus

Skills in practice: safety and sourcing

A Skill is just text, a folder anyone can write and anyone can share. Before you load someone else's, review its source, the same way you would an open-source dependency. Community directories already index over a million skills scraped from GitHub, scale arrived well before quality control did.
In one 2026 audit (Snyk's ToxicSkills study), roughly a third of scanned community skills had at least one security flaw, exposed secrets, insecure credential handling, or prompt-injection-prone instructions. The ecosystem is starting to respond: NVIDIA and others now offer signed, verified skills using the OpenSSF's open signing standard, the same trust model as signed software packages.
Explore live: skills.sh and skillsmp.com, both public skills directories with hundreds of thousands of entries, useful for browsing patterns, review the source before you install anything.
