Act 2 · The headliner

Specialisation
beats intelligence.

The mental model

You already know this pattern

1 Manager→Many Specialists
1 Agent→Many Sub-Agents
No company hires one person to be the security expert, QA, architect and tech writer on the same afternoon. We hire specialists and delegate. So why ask one agent to do all of it in a single chat window, where the security review drowns under test logs and debug chatter?
In Antigravity 2.0, the main agent dynamically defines and invokes sub-agents: clean context each, running in parallel, reporting back only the result.
The debate

Why specialisation wins

Focused contextA specialist sees only what its job needs. Less noise in, better output out. "A focused context beats a stuffed one." Told you it would come back.
Isolated failureOne specialist can fall over while the other three still deliver. One genius fails completely; four specialists fail gracefully. (Demo in Act 3.)
Right-sized modelsSpecialists run on fast, cheap models: Gemini 3.5 Flash outperforms 3.1 Pro while running ~4× faster (Google, I/O 2026). And SkillsBench found smaller models with Skills match larger models without. Specialisation is literally a substitute for raw intelligence.
Isolated workspaces, automaticallyWhen the main agent delegates to sub-agents that need isolation, Antigravity autonomously provisions Git worktrees for them and cleans up afterwards. Your working directory stays untouched. Nobody else does this.

Delegation patterns that work today

Review + fix

The security pair

One sub-agent reviews the diff for vulnerabilities while another drafts the fix. By the time the review lands, the patch is waiting.

Fan-out fetch

One specialist per source

A fetcher each for Jira, GitLab, Confluence, Gmail. Faster aggregation, graceful failure. I run this daily; you will see it live in Act 3.

See like a user

Browser sub-agent QA

It opens your app in its own Chrome profile, clicks through the flow you just changed, and records the video into the walkthrough.

Ship + document

Docs in parallel

Main agent writes the feature; a docs sub-agent writes the README, changelog and ADR alongside. Documentation stops being "later".

Big refactors

Scanner → codemod → verifier

Each in its own auto-provisioned worktree. Three small contexts beat one enormous, confused one.

Compare anything

One researcher per option

Evaluating libraries or vendors? One sub-agent per candidate, merge the findings. Parallel research, clean synthesis.
